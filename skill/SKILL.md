# code-flow-map

把用户问的那段代码的**详细业务逻辑 + 数据脉络**变成一张交互式图：流程骨架可一眼看全，
每个关键节点点开能看清它吃什么、吐什么、数据从哪来到哪去。解决静态图「一张画不下」的痛点。

## 你产出的是什么

一个 **JSON spec**（语义层：节点 / 边 / 分组 / 每节点 I/O / 数据流向），交给打包脚本渲染成
一个**自包含 HTML**。你只负责把代码读懂、抽成 spec —— **绝不要自己算坐标、写 HTML/CSS/JS**。
布局（dagre 自动分层）、手绘渲染（rough.js）、点击右侧面板全在模板里做好了。

## 工作流

### 1. 定位入口
从用户的话里拎出目标（某个函数名 / 命令 / 接口路径 / 模块），用 grep / 读文件找到入口：
路由注册处、同名函数、命令定义。确认起点再往下走。

### 2. 追踪逻辑（核心，决定图的质量）
顺着调用链走一遍，识别四样东西：

- **clusters（分组）**：逻辑阶段，对应图里的大框（如「接入校验」「落库履约」「鉴权」）。
- **nodes（节点）**：有意义的步骤——转换、判定、IO、关键数据产物。抓**业务骨架**，
  不是把每个 helper 都画上；一个节点 = 用户脑子里会单独记一笔的一步。
- **edges（边）**：控制流 + 数据流，标注边上流动的是什么数据。**主数据流**保持默认实线；
  把**异常/错误/回退/降级**边标 `kind:"error"`、**辅助/复用/日志**等次要边标 `kind:"secondary"`
  ——它们默认隐藏（开关可调出），这样主流程是干净骨架，不被一堆跨级错误边糊成一团（长管线尤其重要）。
- **每个节点的 I/O**：去读**真实代码**确定结构——struct/类型定义、JSON 构造处、
  库表 schema、文件格式。**严禁编造字段**。每个 I/O 标 `from`/`to` 指向上下游节点，
  这就是「数据流向」；每个节点带 `source: "file.ts:行号"` 方便溯源。

忠实 > 好看。字段拿不准就去读定义，别猜。关键产物带上真实示例值（像
`{ status: "paid", amount_cents: 3998 }` 这种），用户一看就懂。

### 3. 写 spec
按 `references/spec-schema.md` 的结构产出一个 `<name>.json`。先读那个文件确认字段。
要点：节点 id 唯一；`cluster` 指向已声明的 cluster；I/O 的 `from`/`to` 和 `edges` 的
`from`/`to` 都必须指向存在的节点（脚本会校验，指错会报错）。

### 4. 渲染
```bash
python3 <skill>/scripts/build_flow.py <name>.json <name>-flow.html
```
脚本会校验 spec、内联 dagre+rough+数据成单文件、并自动打开。校验报错就按提示修 spec。
输出 HTML 默认写到临时目录（如 `/tmp`）——**别往被分析的仓库里塞产物**；用户想留存再问放哪。

### 5. 交付说明
告诉用户文件位置 + 怎么读：拖拽平移 / 滚轮缩放 / 点节点看右侧 I/O / 顺「来源·去向」链接追数据。

## 节点 kind 速查

`start` `end` 起止 · `process` 处理/转换 · `decision` 分支判定 · `data` 关键数据产物 ·
`io` 外部 IO（落盘、网络、LLM 调用）。kind 只影响配色/描边，选最贴切的即可。

## I/O 结构的几种 format

按数据实际形态选（一个节点的不同 I/O 可混用）：
- `json` —— 配 `schema`（字段→类型，渲染成可折叠结构树）和/或 `example`（真实示例值）
- `table` —— 配 `columns: [{name,type,desc}]`，渲染成表格（库表/列 schema 最合适）
- `code` —— 配 `example`/`text`，等宽展示（如类型定义、接口签名）
- `text` —— 纯文本说明
- `file` —— 配 `path`，表示一个落盘文件

完整字段、嵌套写法、完整实例见 `references/spec-schema.md` 和
`references/example.json`（可直接当模板改）。

## 心法

用户的出发点是**看懂业务流程 + 摸清数据脉络**。所以：骨架要清晰（分组+主干一眼可循），
数据要可追（每个 I/O 都标来源/去向，让用户能在图里"跟着数据走"），结构要真（宁可去多读
几行代码，也不要给一个看着对、其实字段是编的面板）。

## 重构对比模式（refactor compare）

当用户想**重构/重新设计**一条链路时，产出**旧架构↔新架构双图对比**：左旧右新两张独立图，
选中一侧节点，另一侧对应节点高亮（来源蓝边 → 自动改成 `--map` 绿边表示"对应"），点开看
变更类型 + 重构理由，外加一份迁移计划。**硬约束「不互相污染」**：旧 spec 原封不动、永不
引用新；对应关系只挂在新节点上（`maps_from`），依赖单向 新→旧。

### 工作流

1. **先有旧图**：若还没追踪过目标链路，先按上面的流程产出 `X.old.json`（就是普通 flow spec）。
   新架构没出来之前，可以只给用户看这张旧图。
2. **问清 AI 的参与方式**（三选一，必须问，别擅自决定）：
   - **AI 提方案 + 融合用户想法**：先基于代码主动提 1~N 个重构方向，再结合用户想法收敛。
   - **用户主导，AI 落地**：用户给重构思路/约束，AI 只忠实落成新 spec + 映射，不自作主张。
   - **AI 提多个候选方案**：给几套各带取舍的架构，用户挑一个或组合，再细化。
3. **设计新架构**，写 `X.new.json`（结构同 flow spec，外加）：
   - 每个新节点带 `maps_from:[旧节点id…]`、`change:kept|renamed|merged|split|new|moved`、`rationale`。
     一对多=拆分(多个新节点 maps_from 同一旧节点)；多对一=合并(一个新节点 maps_from 多个旧)；
     `new`=从零新增(`maps_from` 空)。
   - 顶层 `migration:{strategy:"rewrite"|"in-place", summary, steps:[], risks:[]}`——
     给出「从零重写 vs 原地改造」的判断、迁移步骤、风险点（用户要的「认真设计」就在这）。
   - 被删的旧节点放顶层 `removed:[{old_id, rationale}]`。
   - **不要改 `X.old.json`**，新架构既可在旧结构上 copy 改、也可从零设计，但两份文件物理隔离。
4. **渲染**：`python3 scripts/build_flow.py --old X.old.json --new X.new.json X-compare.html`
   （脚本会校验 `maps_from`/`removed.old_id` 都指向真实旧节点，指错报错）。
5. 交付：告诉用户怎么读（选节点看对应高亮、点「源自/去向」跨图跳转、右上「迁移计划」看策略步骤风险）。

字段详见 `references/spec-schema.md` 的「重构对比」一节。
